# PRD 작성 가이드 (Ralph Loop 호환)

이 형식을 따르면 `/wi:start`가 자동으로 WI 체크리스트를 생성하고 Ralph Loop을 가동합니다.

## 필수 섹션

### 1. 프로젝트 개요 (L0)
```markdown
## 프로젝트 개요
- **이름**: 프로젝트명
- **목표**: 이 프로젝트가 해결하는 핵심 문제 (1~2문장)
- **대상 사용자**: 누가 쓰는가
- **성공 기준**: 측정 가능한 목표 (예: 응답시간 200ms 이하)
```

### 2. 기술 스택 (필수!)
```markdown
## 기술 스택
- **언어**: TypeScript / Python / Rust / Go / ...
- **프레임워크**: Next.js / FastAPI / Actix / ...
- **DB**: PostgreSQL / MongoDB / SQLite / ...
- **인프라**: Vercel / AWS / Docker / ...
- **테스트**: Jest / Pytest / ...
- **기타 도구**: Redis, S3, Stripe 등
```
> ⚠️ 이 섹션이 없으면 `/wi:start`의 MCP 자동 탐색 정확도가 낮아집니다. 없어도 동작은 하지만, 명시하면 최적의 도구를 자동 설치합니다.

### 3. 대분류 (L1 - Domain)
```markdown
## 대분류

### 인증 시스템
사용자 인증 및 권한 관리 전반

### 상품 관리
상품 CRUD 및 카테고리 체계

### 주문/결제
주문 처리 및 결제 연동
```

### 4. 중분류 (L2 - Module)
```markdown
### 인증 시스템

#### 회원가입 모듈
이메일/소셜 로그인 회원가입

#### 로그인 모듈
JWT 기반 인증, 토큰 갱신

#### 권한 관리 모듈
역할 기반 접근 제어 (RBAC)
```

### 5. 소분류 (L3 - Feature)
```markdown
#### 회원가입 모듈

##### 이메일 회원가입
- 이메일/비밀번호 입력 폼
- 이메일 인증 발송
- 비밀번호 유효성 검증

##### 소셜 로그인
- Google OAuth 연동
- Kakao 로그인 연동
```

### 6. 상세분류 (L4 - Task) ← 가장 중요!
```markdown
##### 이메일 회원가입

###### 태스크
1. **회원가입 API 엔드포인트 생성**
   - POST /api/auth/register
   - 입력: email, password, name
   - 출력: user object + JWT
   - 검증: 이메일 형식, 비밀번호 8자 이상, 중복 체크

2. **회원가입 폼 UI 구현**
   - 이메일/비밀번호/이름 입력 필드
   - 실시간 유효성 검사
   - 에러 메시지 표시
   - 성공 시 메인 페이지 리다이렉트

3. **이메일 인증 기능**
   - 인증 토큰 생성 및 DB 저장
   - 인증 메일 발송 (템플릿)
   - GET /api/auth/verify?token=xxx
```

## L4 태스크 작성 규칙

| 규칙 | 설명 |
|------|------|
| **크기** | 1개 태스크 = 1개 컨텍스트 윈도우에서 완료 가능 (파일 5개 이내 수정) |
| **명확성** | 입력/출력/검증 기준을 구체적으로 명시 |
| **독립성** | 가능한 한 다른 태스크와 독립적으로 구현 가능 |
| **검증 가능** | "~가 동작한다"로 끝나는 수용 기준 포함 |
| **순서** | 의존성 순서대로 나열 (DB 스키마 → API → UI) |
| **E2E 금지** | E2E/Playwright 테스트 작성은 WI에 포함하지 않음 (아래 참조) |

> **⚠️ E2E 테스트를 WI에 넣지 마세요.**
> `claude -p` 워커는 브라우저를 띄울 수 없어 실제 UI 셀렉터를 확인할 수 없습니다.
> PRD/코드에서 셀렉터를 추측하면 거의 전부 실패합니다.
> E2E 테스트는 대화형 세션에서 Playwright로 실제 화면을 보며 작성하세요.
> 단위 테스트(jest/vitest)는 WI에 포함 가능합니다.

### 좋은 태스크 예시 ✅
```
**사용자 목록 API 구현**
- GET /api/users (페이지네이션)
- 입력: page, limit 쿼리 파라미터
- 출력: { users: [], total: number, page: number }
- 검증: 인증된 관리자만 접근 가능
- 수용 기준: GET /api/users?page=1&limit=10 호출 시 사용자 목록 반환
```

### 나쁜 태스크 예시 ❌
```
**사용자 관리 기능 구현**
- 사용자 CRUD 전체 구현
- UI도 같이 만들기
```
→ 너무 크고 모호함. 쪼개야 합니다.

## 와이어프레임 (필수)

`/wi:prd` Step 3.5에서 자동 생성됩니다.

```
wireframes/
├── index.html              ← 전체 페이지 목록 + 링크
├── dashboard.html           ← 대시보드 와이어프레임
├── employee-list.html       ← 직원 목록
├── attendance.html          ← 출퇴근 기록
└── ...
```

| 규칙 | 설명 |
|------|------|
| **형식** | HTML + 최소 inline CSS (레이아웃 확인용) |
| **data-testid** | 모든 인터랙티브 요소에 data-testid 속성 필수 |
| **구조** | 네비게이션, 테이블, 폼, 버튼, 모달 등 UI 구조 반영 |
| **확정** | 사용자 피드백 → 수정 → 확정 (스킵 불가) |
| **참조** | 워커가 UI 구현 시 와이어프레임의 구조와 data-testid를 따라야 함 |

> 와이어프레임은 PRD 확정 전에 반드시 사용자 승인을 받아야 합니다.
> 워커가 구현할 때 이 와이어프레임을 기준으로 UI를 만들고,
> 대화형에서 E2E 테스트를 작성할 때 data-testid를 셀렉터로 사용합니다.

## 선택 섹션 (있으면 더 좋음)

### 비기능 요구사항
```markdown
## 비기능 요구사항
- 응답시간: API 95th percentile < 200ms
- 가용성: 99.9% uptime
- 보안: OWASP Top 10 대응
- 접근성: WCAG 2.1 AA 준수
```

### 외부 연동
```markdown
## 외부 연동
- Stripe: 결제 처리 (API key 필요)
- SendGrid: 이메일 발송
- S3: 파일 업로드
```
> 외부 연동이 명시되면 해당 MCP 서버를 자동으로 검색합니다.

### 마일스톤
```markdown
## 마일스톤
1. Phase 1 - MVP: 회원가입 + 로그인 + 기본 CRUD
2. Phase 2 - 고도화: 결제 연동 + 알림
3. Phase 3 - 운영: 모니터링 + 성능 최적화
```

## Boundaries

**Will:**
- PRD 작성 가이드 및 템플릿 출력

**Will Not:**
- PRD 직접 작성 (사용자가 작성해야 함)
- 프로젝트 환경 변경
